Agent basique + 1 Tool

Fondation
📁

Structure obligatoire

mon_agent/
  __init__.py   # from . import agent
  agent.py      # root_agent = Agent(...)
  .env          # GOOGLE_API_KEY=...
🤖

root_agent

Variable obligatoire qu'ADK cherche. Agent = alias de LlmAgent.

root_agent = Agent(
  name, model, description,
  instruction, tools
)
🔧

Tool = fonction Python

Nom = nom du tool. Docstring = description pour le LLM. Type hints = schéma JSON.

Retour préféré : dict avec "status"

💡

Règle d'or (mock tools)

Un tool mock qui prétend générer du contenu mais retourne toujours la même chose confond le LLM. Dans un tuto : le tool sauvegarde, le LLM génère. En production, un tool peut générer du contenu réel (API, DB, etc.).

2

Plusieurs Tools

Sélection LLM
🎯

Le LLM choisit

Le LLM reçoit le schéma de tous les tools et sélectionne le plus pertinent selon la requête.

📝

Docstring = guide de sélection

Des docstrings vagues = routing ambigu. Des docstrings spécifiques = choix précis.

⚡

Tool sans paramètre

def list_templates() -> dict est valide. Le LLM l'appelle quand c'est pertinent.

🔗

Chainage dans un tour

Le LLM peut appeler plusieurs tools dans le meme tour si la requete le demande.

Runtime, Config & Données

Production
🌡️

generate_content_config

temperature=0.2 = précis.
temperature=0.9 = créatif.
max_output_tokens = limite.

GenerateContentConfig(
  temperature=0.2,
  max_output_tokens=500
)
📐

output_schema (Pydantic)

Force le LLM à retourner du JSON structuré.

class Analysis(BaseModel):
    score: int
    issues: list[str]

Agent(output_schema=Analysis)
📎

Artifacts vs State

State = key-value texte.
Artifacts = fichiers binaires versionnés.

await ctx.save_artifact("report.pdf", part)
await ctx.load_artifact("report.pdf")
🧠

Memory vs State

State = 1 session.
Memory = cross-sessions, recherche sémantique.

from google.adk.tools import load_memory
Agent(tools=[load_memory])
🏭

Runner : exécution programmatique

adk web = dev. Runner = production.

runner = Runner(agent=root_agent, app_name="app", session_service=InMemorySessionService())
async for event in runner.run_async(user_id="u1", session_id="s1", new_message=...):
    if event.is_final_response(): print(event.content.parts[0].text)
4

Session State & ToolContext

Mémoire
🔌

ToolContext

Ajouter tool_context: ToolContext comme dernier paramètre. Auto-injecté par ADK, invisible pour le LLM.

def my_tool(x: str, tool_context: ToolContext):
    tool_context.state["key"] = val
🏷️

Préfixes de state

(aucun)  → session courante
user:    → toutes sessions user
app:     → global
temp:    → invocation seule ⚠️
📥

output_key

Sauvegarde auto la réponse texte de l'agent dans le state.

Agent(output_key="result")
# → state["result"] = reponse
📤

{state_var} templating

Les {clés} dans l'instruction sont remplacées au runtime par les valeurs du state.

instruction="Theme: {topic}"
# state["topic"]="Python" → "Theme: Python"

Délégation Multi-Agent

Équipe
👔

Coordinateur

Pas de tools. Route les requêtes aux spécialistes via sub_agents.

Agent(sub_agents=[a, b, c])
# PAS de tools=[]
🏷️

description = routing

Le LLM du coordinateur lit le description de chaque sub-agent pour décider. Sois spécifique.

↗️

transfer_to_agent

Généré automatiquement par le LLM. Tu n'as rien à coder.

# Le LLM fait tout seul :
transfer_to_agent("code_writer")
🔧

AgentTool

sub_agents = transfère le contrôle.
AgentTool = appel comme un tool, le parent garde le contrôle.

from google.adk.tools import AgentTool
tools=[AgentTool(agent=specialist)]
User
→
Coordinator
→
Writer
Reviewer
Greeter

sub_agents: le LLM transfère le contrôle  |  AgentTool: le parent appelle et garde le contrôle

Coordinator
→ tool call →
Explainer (AgentTool)
→ result →
Coordinator répond
6

SequentialAgent

Pipeline
➡️

Ordre fixe, déterministe

Pas de model, pas d'instruction. Execute les sub-agents dans l'ordre de la liste. Toujours le meme flux.

🔄

Data flow

Agent N ecrit via output_key, Agent N+1 lit via {state_var}.

writer:     output_key="code"
reviewer:   instruction="{code}"
⚖️

Sequential vs Délégation

Sequential = ordre fixe, pas de LLM pour router.
Délégation = le LLM choisit dynamiquement.

Writer
→ state["code"] →
Reviewer
→ state["feedback"] →
Refactorer
→ state["final"]
7

ParallelAgent

Concurrent
⚡

Exécution simultanée

Tous les sub-agents demarrent en meme temps. L'ordre dans la liste est irrélevant.

🚫

Pas de partage de state

Pendant l'exécution, les branches ne voient PAS les changements des autres. Les output_key sont mergés après.

🧱

Pattern classique

SequentialAgent([
  writer,
  ParallelAgent([a, b, c]),
  synthesizer
])
Writer
→
Parallel
Security
Performance
Style
→
Synthesizer
8

LoopAgent

Itération
🔁

Boucle création/critique

Execute ses sub-agents en sequence, puis recommence. Parfait pour l'amélioration itérative.

🛑

2 façons de sortir

max_iterations = sécurité anti-boucle infinie.
escalate = True = sortie anticipée (le reviewer décide).

tool_context.actions.escalate = True
💾

State persiste entre itérations

Le feedback de l'itération N est lisible en N+1. C'est ce qui permet l'amélioration progressive.

Loop
Writer
→
Reviewer
score<8 → ↩ retry
score≥8
ESCALATE ✓
9

Callbacks

Guardrails
🛡️

before_model_callback

Intercepte avant le LLM. Bloque les prompts dangereux.

(Context, LlmRequest)
  → None (passe)
  → LlmResponse (bloque)
🔍

before_tool_callback

Valide les arguments avant l'exécution du tool.

(BaseTool, dict, Context)
  → None (passe)
  → dict (bloque)
📋

after_model / after_tool

Post-traitement : logger, filtrer, modifier les réponses. Même logique : None = garder, valeur = remplacer.

🔑

Règle universelle

return None = laisser passer.
Retourner une valeur = bloquer/remplacer.
Les callbacks sont transversaux (un seul protege tout).

Concepts avancés

Expert
🧬

BaseAgent

Logique d'orchestration arbitraire. Surcharger _run_async_impl pour des patterns non-standard (conditions, routing dynamique).

⏳

LongRunningFunctionTool

Pour les ops longues (approbation humaine). Retourne "pending", l'agent pause.

LongRunningFunctionTool(func=my_fn)
🔐

Plugins sécurité

Gemini-as-Judge (prompt injection), PII Redaction (données perso), Model Armor. Transversaux (protègent tout).

📊

Evaluation

adk eval agent/ eval_set.json

Tests systématiques avec fichier JSON de cas de test.

🌊

Streaming + MCP + OpenAPI + A2A

Streaming (SSE), MCP (protocol standard), OpenAPI (API REST), A2A (inter-systemes). Pour la production avancée.
